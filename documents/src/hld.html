1. System Overview

PENUMBRA is a two-tier web system — a Python forecasting/orchestration backend and a React charting frontend — arranged around a cache-first data pipeline. Public NOAA/GFZ feeds are polled on schedule, normalised into aligned time series, run through a leakage-free walk-forward backtest to learn per-lead-time error distributions, and then combined with a point model to emit probabilistic forecasts with verified calibration. The pipeline is modelled as six named agents; the same graph is rendered as the n8n-style diagram embedded in the UI.

3. Forecasting Approach (rationale)

PENUMBRA deliberately uses a transparent, defensible statistical model rather than an opaque black box, because the deliverable is calibrated uncertainty, which must be verifiable:

  • F10.7 point: F10.7 is strongly autocorrelated day-to-day and modulated by the 27-day solar-rotation period. The point forecast blends short-lag persistence with mean-reversion toward the trailing 27-day mean and the longer cycle level, the reversion strength increasing with lead time. This matches the physical behaviour and is a strong, honest baseline.
  • Uncertainty: the band at lead L is the point forecast plus the empirical p05…p95 of the out-of-sample forecast errors at lead L, measured by walk-forward backtest. Because the band is learned from real forecast errors, it is calibrated by construction if the backtest is leakage-free — which is the single most important correctness property (CON-2, FR-VER-1).
  • Kp: a bounded quasi-discrete index; a point value is misleading. PENUMBRA emits category probabilities from recency-weighted historical transition frequencies conditioned on the current activity level — an operationally meaningful "chance of a storm" per day.
  • Skill: every claim is checked against NOAA's own 45-day forecast, so "better than the official product" is a measured, reproducible statement, not a marketing one.
Honest scope. This is a statistically calibrated reduced-model forecaster, not a physics ensemble. The value is that it publishes verified uncertainty where the operational products publish none — exactly the gap identified by Parker et al. (2024).

4. Data Design (summary)

SQLite (backend/app/data/cache.db): kv_cache (source payloads + TTL), forecast_snapshots (issued forecast + calibration read-model as JSON documents), and pipeline_runs (agent, timing, status, items, error). The CacheService interface is the single seam for a later move to managed storage.

5. Error Handling & Logging

  • Exception hierarchy rooted at PenumbraError(code, message, http_status): DataSourceError (502), InsufficientDataError (422), ForecastError (500), NotFoundError (404), ConfigError (500), ValidationFailure (422).
  • Boundary rule: logic raises; service contextualises & decides fallback (cache, demo, skip); API converts to the JSON error envelope. The UI never sees a stack trace.
  • Structured logging (request_id middleware); agent runs logged start/finish with duration and item counts. WARN = degraded (fallback used), ERROR = user-visible failure.

7. Deployment View

v1.0: localhost (uvicorn :8001 + Vite :5175 proxy) and the shared Lightsail VPS (13.127.244.0). On the VPS, PENUMBRA runs as its own systemd service on port 8001, exposed through an isolated nginx server block on a dedicated public port, with KESSLER's port-80 default server left entirely untouched (NFR-5). A bounded MemoryMax and the host's existing swap keep the two services co-resident on 414 MB. v2.0: managed storage/scheduler behind the CacheService seam; no logic-layer change anticipated.
