Product: PENUMBRA — Probabilistic Space-Weather Driver Forecasting

1. Introduction

1.1 Purpose

This SRS defines the functional and non-functional requirements for PENUMBRA, an open-source system that produces probabilistic forecasts of the space-weather drivers that govern satellite orbital drag — the F10.7 solar radio flux and the Kp geomagnetic index — attaches calibrated uncertainty to every prediction, verifies that calibration against ground truth, and translates the driver uncertainty into orbital-drag and orbit-decay risk. It is delivered as a REST API and a web dashboard.

1.2 Scope

Atmospheric drag is the largest source of orbit-prediction error in LEO, and the dominant sub-source is frequently the forecast of the drivers feeding the density model, not the density model itself (Parker et al., 2024). Operational driver forecasts — including NOAA's 45-day F10.7/Ap product — are published as point values with no uncertainty quantification, forcing operators to make unsupported assumptions during orbit and conjunction assessment. Research-grade probabilistic methods exist but there is no operational product. PENUMBRA fills that gap. It is the phase-2 companion to the KESSLER conjunction-assessment toolkit; together they form an open orbital-risk stack.

Out of scope for v1.0: full physics-based thermosphere modelling (PENUMBRA uses a documented reduced density model for the impact translation), ionospheric/TEC forecasting, solar-flare prediction, and paid data sources.

1.3 Definitions and Acronyms

TermDefinition
F10.7Solar radio flux at 10.7 cm (2800 MHz), in solar flux units (sfu). Primary proxy for solar EUV heating of the thermosphere.
KpPlanetary geomagnetic activity index, quasi-logarithmic scale 0–9 (in thirds), 3-hourly.
ApDaily linearised geomagnetic amplitude index derived from Kp.
DriverAn input index (F10.7, Kp/Ap) that drives an atmospheric-density model.
Lead timeForecast horizon: days ahead of issue date (1…45).
Quantile bandInterval between two forecast quantiles (e.g. p05–p95 = 90% band).
CalibrationProperty that a stated X% band contains the truth X% of the time over many forecasts.
CoverageEmpirical fraction of observations falling inside a stated band.
Pinball lossProper scoring rule for a quantile forecast (a.k.a. quantile loss).
Skill scoreRelative improvement of a forecast over a reference (here, NOAA's 45-day forecast).
SWPCNOAA Space Weather Prediction Center.

2. Overall Description

2.1 Product Perspective

PENUMBRA is a self-contained web application plus a reusable forecasting core. It consumes only free public NOAA/GFZ data and requires no paid dependency. It runs fully on a small VPS; the service layer isolates storage and scheduling behind interfaces so the system can later move to cloud infrastructure without touching the forecasting logic.

2.2 User Classes

User classDescriptionPrimary interest
Satellite operatorPlans manoeuvres and orbit maintenance."How uncertain is my predicted orbit over the next N days, and why?"
Orbit-prediction / SSA softwareConsumes driver forecasts programmatically.A driver distribution, not a point, to propagate state uncertainty honestly.
Space-weather researcherStudies driver forecasting and calibration.Reproducible probabilistic forecasts, verification metrics, skill vs NOAA.
Demo viewerEvaluates the project.Clear fan charts, the calibration story, polish.

2.3 Operating Environment

  • Backend: Python ≥ 3.11 (3.9 supported via typing back-port), FastAPI/uvicorn.
  • Frontend: Node ≥ 18, Vite build, evergreen browsers.
  • v1.0 target: localhost and a shared Lightsail VPS (co-hosted with KESSLER, isolated service/port).

2.4 Design Constraints

CON-1Forecasts MUST carry calibrated uncertainty; a point value without a band is not an acceptable output. Uncertainty is the product.
CON-2Uncertainty bands MUST be derived from a leakage-free walk-forward backtest (out-of-sample error quantiles), never fit in-sample.
CON-3The system MUST be fully functional with zero credentials (Demo Mode) using bundled historical data.
CON-4NOAA/GFZ are public but not to be hammered: cache-first architecture, polite intervals; no per-user live fetches.
CON-5Repository layout mirrors KESSLER: separate ui/ and backend/; backend separates logic/ (pure math) and service/ (I/O). Deploy MUST NOT disturb the co-hosted KESSLER service.

3. Functional Requirements

3.1 Data Ingestion (FR-ING)

IDRequirementPriority
FR-ING-1SHALL ingest daily observed F10.7 from NOAA SWPC (f107_cm_flux.json) and long-history monthly indices (solar-cycle/observed-solar-cycle-indices.json), keyless.Must
FR-ING-2SHALL ingest planetary Kp from NOAA SWPC (products/noaa-planetary-k-index.json) and derive daily max-Kp / Ap.Must
FR-ING-3SHALL ingest NOAA's official 45-day F10.7/Ap forecast (45-day-forecast.json) as the reference baseline for skill scoring.Must
FR-ING-4All external fetches SHALL be cached with per-source TTL (observations ≥ 3 h, forecast ≥ 6 h) and served from cache within TTL.Must
FR-ING-5In Demo Mode the system SHALL load bundled historical series and mark responses data_mode: "demo".Must
FR-ING-6Ingestion failures SHALL degrade to last-known-good cache with a structured warning, never a hard failure.Must

3.2 F10.7 Probabilistic Forecast (FR-F107)

IDRequirementPriority
FR-F107-1SHALL produce a point forecast of daily F10.7 for lead times 1–45 days using a persistence-with-mean-reversion model toward the local solar-rotation (27-day) and cycle climatology.Must
FR-F107-2SHALL attach quantile bands (at least p05, p25, p50, p75, p95) per lead day, derived from the empirical distribution of out-of-sample forecast errors at that lead time (FR-VER-1).Must
FR-F107-3Bands SHALL be monotonic (p05 ≤ p25 ≤ … ≤ p95) and F10.7 SHALL be floored at the physical quiet-Sun minimum (~64 sfu).Must
FR-F107-4All forecast math SHALL be pure functions in the logic layer (no I/O) and unit-tested, including monotonicity and a synthetic-signal recovery test.Must

3.3 Kp / Geomagnetic Forecast (FR-KP)

IDRequirementPriority
FR-KP-1SHALL produce, per lead day 1–7, probabilities across geomagnetic categories: Quiet (Kp<3), Unsettled (3≤Kp<4), Active (4≤Kp<5), Storm (Kp≥5).Must
FR-KP-2Category probabilities SHALL be estimated from the recency-weighted historical transition frequencies conditioned on recent activity, and SHALL sum to 1 per lead day.Must
FR-KP-3SHALL expose the expected daily Ap with an uncertainty band for downstream drag modelling.Should

3.4 Verification & Calibration (FR-VER)

IDRequirementPriority
FR-VER-1SHALL run a walk-forward backtest over the historical record: at each origin, forecast forward, then record the signed error per lead time — with a strict train/verify time split (no future leakage).Must
FR-VER-2SHALL compute band coverage (empirical fraction inside p05–p95 and p25–p75) per lead time; a calibrated 90% band targets 0.90 coverage.Must
FR-VER-3SHALL compute a reliability diagram (predicted vs observed frequency) for the Kp categorical forecast.Must
FR-VER-4SHALL compute pinball loss for the F10.7 quantiles and a skill score of the PENUMBRA point forecast against NOAA's 45-day forecast (RMSE-based, per lead time).Must
FR-VER-5The calibration report SHALL be exposed via API and rendered in the UI — it is the product's primary credibility artifact.Must

3.5 Drag-Impact Translation (FR-DRAG)

IDRequirementPriority
FR-DRAG-1SHALL map the F10.7 forecast band to a thermospheric density band at reference altitudes (e.g. 400, 550, 800 km) via a documented reduced density model (F10.7-scaled exponential atmosphere).Must
FR-DRAG-2SHALL express the resulting along-track position / orbit-decay uncertainty growth for a nominal LEO object over the forecast window — the explicit bridge to conjunction assessment.Should
FR-DRAG-3SHALL clearly label the drag translation as reduced-model, order-of-magnitude guidance — not a high-fidelity propagation.Must

3.6 API (FR-API)

IDRequirementPriority
FR-API-1REST API under /api/v1: F10.7 forecast, Kp forecast, drag impact, calibration report, recent observations, pipeline status, health.Must
FR-API-2All errors SHALL use one envelope {"error":{code,message,detail,request_id}} with correct HTTP status.Must

IDRequirementPriority
FR-UI-1The dashboard SHALL render an F10.7 fan chart: observed history, point forecast, and nested p25–p75 / p05–p95 bands over 45 days, with NOAA's forecast overlaid for comparison.Must
FR-UI-2SHALL render the Kp storm-probability forecast as a stacked per-day category strip, and the drag-impact panel.Must
